# Reliquary SN81 — Prompt Dataset Analysis

Analysis of the datasets that supply **prompts for GRPO training / validation** on
Bittensor subnet 81 (Reliquary). Covers both active RL environments:
**OpenMathInstruct‑2** and **OpenCodeInstruct**.

> Scope: where prompts come from, how each environment buckets / selects them,
> counts, answer formats, length & token use, reward computation, and the
> configuration knobs. All numbers are traced to source files (see
> [Provenance](#provenance)).
>
> Repo layout referenced here:
> - Protocol code: `../reliquary/` (the `reliquary` package)
> - Miner eval + diagnostics: `v2-35-h200-5-working/sn81-miner/`

---

## TL;DR

| | **OpenMathInstruct‑2** | **OpenCodeInstruct** |
|---|---|---|
| Env name | `openmathinstruct` | `opencodeinstruct` |
| Upstream | `nvidia/OpenMathInstruct-2` (~14M) | `nvidia/OpenCodeInstruct` (~5M) |
| Loaded pool | first **2 of 32** shards = **873,300** rows | filtered subset, **~50,000** rows |
| **Source buckets** | **4** (`problem_source` column) | **none** — categorical columns stripped |
| Task | math word/competition problems | Python code generation |
| Reward | string compare on last `\boxed{}` | `passed/total` over hidden unit tests |
| Reward authority | miner‑claimed, validator‑verified | **validator‑authoritative** (hidden cases) |
| Default enabled? | **yes** (`DEFAULT_ENVIRONMENTS`) | opt‑in (sandbox/runsc coordination) |
| Selection knob | `RELIQUARY_PROMPT_SOURCES` | none (no source dimension) |

Both run side‑by‑side in `ENVIRONMENT_MIX` (8 prompts each per optimizer step →
16 prompts × 8 rollouts = 128 sequences), but the CLI/Docker default is
OpenMath‑only.

## 1. Environment mix & selection

Defined in [`../reliquary/reliquary/constants.py`](../reliquary/reliquary/constants.py):

```python
ENVIRONMENT_MIX = [
    ("openmathinstruct", B_BATCH),   # 8 prompts / step
    ("opencodeinstruct", B_BATCH),   # 8 prompts / step
]
DEFAULT_ENVIRONMENTS = "openmathinstruct"   # OpenCode is opt-in
B_BATCH      = 8     # distinct prompts per env per window
M_ROLLOUTS   = 8     # rollouts per prompt (GRPO group size)
DEFAULT_BASE_MODEL = "Qwen/Qwen3.5-4B"
```

- Prompt **selection** is a deterministic `get_problem(index)` with modulo wrap
  over the loaded rows — the same `index` yields the same prompt across miners,
  which is required for GRAIL consensus.
- Active envs chosen at runtime via `RELIQUARY_ENVIRONMENTS`; OpenCode only
  activates when explicitly included.

## 2. OpenMathInstruct‑2 (`openmathinstruct`)

**Source:** `nvidia/OpenMathInstruct-2` — 14M synthetic math problems generated by
Llama‑3.1‑405B from the MATH + GSM8K seed sets. Pre‑extracted `expected_answer`
makes reward parsing reliable.
Code: [`../reliquary/reliquary/environment/openmathinstruct.py`](../reliquary/reliquary/environment/openmathinstruct.py)

### Loading

Loads only the first **N parquet shards** (default 2 of 32) for disk efficiency:

```python
RELIQUARY_OMI_SHARDS = 2          # env var, clamped to [1, 32]
# data_files = data/train-00000-of-00032.parquet ... (first N)
# split="train", verification_mode="no_checks"
```

2 shards ≈ 873,300 rows ≈ ~1 GB. Each shard ≈ 440k problems / ~500 MB.

### Upstream schema (`nvidia/OpenMathInstruct-2`)

Per the [HF dataset card](https://huggingface.co/datasets/nvidia/OpenMathInstruct-2) — each row:

| Column | Type | Range | Used by env? |
|---|---|---|---|
| `problem` | string | 13 – 13,600 chars | ✓ → `prompt` (+ boxed instruction) |
| `generated_solution` | string | 190 – 50,700 chars | ✗ unused (synthetic CoT solution) |
| `expected_answer` | string | 0 – 534 chars | ✓ → `ground_truth` |
| `problem_source` | categorical (4) | — | drives `RELIQUARY_PROMPT_SOURCES` filtering |

`problem_source` ∈ {`augmented_gsm8k`, `augmented_math`, `gsm8k`, `math`}.
Splits: `train` = **14M** (32 parquet shards — what the env loads), plus
fair‑downsampled `train_1M` / `train_2M` / `train_5M`; `default` ≈ 22M rows.

### The 4 source buckets (`problem_source`)

Measured over the default 2‑shard pool (873,300 rows) —
see [`v2-35-h200-5-working/sn81-miner/diagnostics/source_probe.out`](v2-35-h200-5-working/sn81-miner/diagnostics/source_probe.out):

| Source bucket | Rows (2‑shard) | Share | Description |
|---|---:|---:|---|
| `augmented_math`    | **558,152** | 63.9% | Synthetic variants of competition MATH |
| `math`              | **154,133** | 17.6% | Seed competition MATH problems |
| `augmented_gsm8k`   | **132,489** | 15.2% | Synthetic variants of grade‑school word problems |
| `gsm8k`             | **28,526**  | 3.3%  | Seed GSM8K word problems |
| **Total**           | **873,300** | 100%  | first 2 of 32 shards |

- Shards are a shuffle, so these proportions scale to the full ~14M dataset.
- **GSM family** (`gsm8k` + `augmented_gsm8k`) = **161,015** rows.

**Full `train` split (14M) — extrapolated** (2‑shard counts × 16, assuming uniform shard composition):

| Source bucket | ~Rows (14M `train`) | Share |
|---|---:|---:|
| `augmented_math`    | ~8.93M  | 63.9% |
| `math`              | ~2.47M  | 17.6% |
| `augmented_gsm8k`   | ~2.12M  | 15.2% |
| `gsm8k`             | ~0.46M  | 3.3%  |
| **Total**           | ~13.97M | 100%  |

GSM family ≈ **2.58M**. The env only materializes the first 2 shards by default;
raise `RELIQUARY_OMI_SHARDS` (≤ 32) for more coverage.

### Miner source restriction

The miner restricts the candidate pool to the **GSM family** via
`RELIQUARY_PROMPT_SOURCES=gsm8k,augmented_gsm8k`
([`v2-35-h200-5-working/sn81-miner/bin/setup.sh`](v2-35-h200-5-working/sn81-miner/bin/setup.sh)).
Confirmed in logs:

```
candidate pool: 161015/873300 prompts with problem_source in ['augmented_gsm8k', 'gsm8k']
```

**Why GSM‑only:** answer‑format reliability. The reward parser keys off the last
`\boxed{}` content, so clean integer/decimal answers parse deterministically;
ambiguous LaTeX forms scatter in/out of the parser even when the model is right.

| Bucket | Typical answer formats | Parser behaviour |
|---|---|---|
| `gsm8k`, `augmented_gsm8k` | `plain_int` (~45%), `decimal` (~55%) | clean, reliable |
| `math`, `augmented_math` | fractions, `\sqrt`, `\pi`, matrices, tuples, variables | ambiguous (exploited by the brittle‑pool / format‑ambiguity strategy) |

Per‑profile answer‑format breakdowns:
[`v2-35-h200-5-working/sn81-miner/diagnostics/target_profiles.txt`](v2-35-h200-5-working/sn81-miner/diagnostics/target_profiles.txt).

### Prompt encoding, length & token use

- Each problem gets the answer‑format instruction appended:
  `"\n\nPut your final answer within \boxed{}."`
  ([`openmathinstruct.py`](../reliquary/reliquary/environment/openmathinstruct.py) `_ANSWER_FORMAT_INSTRUCTION`).
- Encoded via the shared canonical encoder
  [`../reliquary/reliquary/protocol/tokens.py`](../reliquary/reliquary/protocol/tokens.py) `encode_prompt()`:
  Qwen3.5 chat template, `add_generation_prompt=True`, **`enable_thinking=False`**
  (long CoT would overrun the token cap before the `\boxed{}` answer).
- Dataprep uses problem **char length** as an "easy/terminating" proxy:
  `≤ 400` chars ([`dataprep/format_analysis.py`](v2-35-h200-5-working/sn81-miner/dataprep/format_analysis.py)) /
  `≤ 500` chars ([`dataprep/brittle_pool.py`](v2-35-h200-5-working/sn81-miner/dataprep/brittle_pool.py)).
- Observed **completion** lengths (model output, accepted in‑zone rollouts):
  GSM‑family miners mean ≈ 430–440 tok (median 446–462, capped ~510); a
  math‑mix miner ran mean 946 / median 830 / max 1960 tok (harder problems).
- Protocol token caps: `max_model_len = 9216`; `MAX_NEW_TOKENS_PROTOCOL_CAP = 8192`.

Field length characteristics (dataset‑wide ranges from the HF card; **not** per‑source):

| Field | Char range |
|---|---:|
| `problem` (→ prompt) | 13 – 13,600 |
| `expected_answer` | 0 – 534 |
| `generated_solution` (unused) | 190 – 50,700 |

Per‑source prompt char/token percentiles (to be filled by `source_stats.py`):

| Source bucket | prob_chars med/p90/max | prompt_tokens med/p90/max | ans_chars med/max |
|---|---|---|---|
| `augmented_math`  | _pending_ | _pending_ | _pending_ |
| `math`            | _pending_ | _pending_ | _pending_ |
| `augmented_gsm8k` | _pending_ | _pending_ | _pending_ |
| `gsm8k`           | _pending_ | _pending_ | _pending_ |

> The token columns require the raw dataset + Qwen3.5 tokenizer, which live on
> the H200 box. Run [`source_stats.py`](#6-tooling) there to populate them
> (it writes `/root/source_stats.json` in exactly this shape).

### Reward

`1.0` if the normalized last `\boxed{}` matches the normalized `expected_answer`,
else `0.0`. Balanced‑brace parser + LaTeX/number normalization; never raises.
A `BOXED_ANSWER_MIN_PROB = 0.001` guard flags final‑answer token tampering.

## 3. OpenCodeInstruct (`opencodeinstruct`)

**Source:** `nvidia/OpenCodeInstruct` — ~5M Python instruction/code pairs with
assertion‑based unit tests (CC BY 4.0, ~6.4 GB).
Code: [`../reliquary/reliquary/environment/opencodeinstruct.py`](../reliquary/reliquary/environment/opencodeinstruct.py)
Design spec: [`../reliquary/docs/superpowers/specs/2026-05-22-opencodeinstruct-env-design.md`](../reliquary/docs/superpowers/specs/2026-05-22-opencodeinstruct-env-design.md)

### No source buckets

Unlike OpenMath, the OpenCode subset has **no categorical source/domain column**.
The build pipeline keeps only `input`, `structured_cases`, `id`; the miner
mirror keeps only `input`, `id`. There is **no `RELIQUARY_PROMPT_SOURCES`
equivalent** for OpenCode.

The upstream `nvidia/OpenCodeInstruct` *does* carry categorical metadata, but the
build **strips everything** except `input` (plus a derived `structured_cases`).
Per the [HF dataset card](https://huggingface.co/datasets/nvidia/OpenCodeInstruct)
(~4.97M rows, CC‑BY‑4.0):

| Column | Type | Range | Kept in subset? |
|---|---|---|---|
| `id` | string | 32 chars | ✓ |
| `input` | string | 116 – 7,620 chars | ✓ → `prompt` |
| `output` | string | 74 – 6,660 chars | ✗ ref solution (used only to validate, then dropped) |
| `domain` | class (2): `generic`, `algorithmic` | — | ✗ stripped |
| `generation_algorithm` | class (2): `self-instruct`, `evol-instruct` | — | ✗ stripped |
| `llm_judgement` | string (JSON) | 364 – 2,070 chars | ✗ |
| `unit_tests` | string (list) | 190 – 9,150 chars | ✗ raw → converted to `structured_cases` |
| `tests_execution_status` | string (list `pass`/`fail`) | — | ✗ |
| `average_test_score` | class (11): 0.0 – 1.0 | — | filter `== 1.0`, then dropped |
| `structured_cases` | *(added by build)* | — | ✓ validator only |

So OpenCode's nearest analog to a "source bucket" is the upstream **`domain`**
(`generic` / `algorithmic`) or **`generation_algorithm`** (`self-instruct` /
`evol-instruct`) — but **both are discarded at build time** and unavailable to
miner or validator. There is no runtime source dimension.

### Selection = quality/determinism filter cascade

Instead of source categories, the pool is defined by offline filters in
[`../reliquary/scripts/build_opencodeinstruct_subset.py`](../reliquary/scripts/build_opencodeinstruct_subset.py):

| Stage | Filter | Effect |
|---|---|---|
| 0 | upstream `nvidia/OpenCodeInstruct` | ~5M rows |
| 1 | `average_test_score == 1.0` (ref solution passes its own tests) | drops ~30% → ~3–4M |
| 2 | `unit_tests` parses as a list of strings | drop on parse fail |
| 3 | only **simple deterministic asserts** convert to structured cases (`assert fn(args)==literal`, reversed, truthy/falsy, `Solution().method(...)`) | drop if 0 convertible |
| 4 | reject non‑deterministic test source (`random`/`time`/`os`/`subprocess`/`threading`/`socket`/`urllib`/…) | deny‑list `_NONDET_PATTERNS` |
| 5 | **double‑execution** check (`PYTHONHASHSEED` 0 vs 1, both pass all cases) | drop flaky/crashing/timeout |
| → | **built & verified at ~50,000 rows** | the actual prompt pool |

The filters bias the distribution toward small, self‑contained algorithmic
functions expressible as `fn(args) == literal`.

### Reward & sandbox

- **Validator‑authoritative**: miners submit placeholder rewards and never see
  hidden cases. The validator recomputes `passed/total` (linear) before zone
  checks, archive, and training.
- Grading runs miner code in a **gVisor (`runsc`) sandbox**: 5 s CPU, 256 MB,
  `--network=none`, tmpfs `/tmp` 10 MB, max 16 procs, 7 s outer wall‑clock.
  Trusted grader owns expected values over a Unix socket; the worker only ever
  receives `(code, entrypoint, args, kwargs)`.
- Any failure (timeout, crash, grader down, empty code) resolves to `0.0` —
  `compute_reward` never raises.

## 5. Gotchas

- **`DATASET_NAME = "karpathy/climbmix-400b-shuffle"`** in `constants.py` is under
  a `DEPRECATED (GRPO REFACTOR)` banner. It is kept importable only so transitive
  imports don't break; it does **not** drive prompt selection. Ignore it.
- There is **no `.env`** in the protocol repo — only `.env.example`. All dataset
  overrides fall back to in‑code defaults.
- `R2_BUCKET_ID=reliquary` in `.env.example` is an **object‑storage** bucket
  (rollout archives), unrelated to dataset "source buckets."
- OpenMath answer‑format ambiguity in the `math`/`augmented_math` buckets is the
  basis of the brittle‑pool exploitation strategy — see
  [`dataprep/brittle_pool.py`](v2-35-h200-5-working/sn81-miner/dataprep/brittle_pool.py).
